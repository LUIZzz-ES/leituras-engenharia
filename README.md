# Notas de leitura — Código Limpo, capítulos 2, 4 e 5

> Escrito por um estudante do 1º semestre de Engenharia de Software.
> Não é resumo acadêmico. É o que ficou na cabeça depois de ler.

---

## Cap. 2 — Nomes com significado

A ideia central: o nome de uma variável, função ou classe precisa revelar o que ela representa — sem precisar de comentário pra explicar.

```java
// o que isso significa?
int d;

// agora ficou claro
int diasDesdeUltimaModificacao;
```

Se você precisa de um comentário pra explicar o nome, o nome está errado.

---

## Cap. 4 — Comentários

Contraintuitivo, mas faz sentido: comentário é sinal de código mal escrito.

Não é que seja proibido comentar. É que, se o código precisa de explicação pra ser entendido, o problema está no código. A solução é reescrever, não comentar por cima.

---

## Cap. 5 — Formatação

Formatação afeta a leitura tanto quanto a lógica. Um arquivo bem organizado tem ritmo — você sabe onde procurar cada coisa.

Pontos principais do capítulo:
- Separe conceitos distintos com linhas em branco
- Mantenha funções relacionadas próximas umas das outras
- Linhas curtas são mais fáceis de ler do que linhas longas e densas
